# AVR (Alf and Vegard's RISC) TWI (Two Wire Interface) / I2C (Inter-Integrated Circuit) — Bare-Metal Driver

Reference for driving I2C devices directly on the ATmega328P without `Wire.h`.
Written from building the BH1750 light sensor driver for this project.

---

## What Is I2C?

I2C (Inter-Integrated Circuit) is a two-wire serial protocol invented by Philips.
Two wires connect a master (the microcontroller) to one or more slaves (sensors, displays, etc.):

| Wire | Name                | Purpose                           |
|------|---------------------|-----------------------------------|
| SDA  | Serial Data         | Bidirectional data line           |
| SCL  | Serial Clock        | Clock generated by the master     |

On Arduino Uno R3:
- **SDA → A4** (PC4)
- **SCL → A5** (PC5)

Both lines are **open-drain** with external pull-up resistors (typically 4.7 kΩ to 3.3V or 5V).
"Open-drain" means any device can pull the line LOW, but no one drives it HIGH — the resistor
does that passively. This lets multiple devices share the same bus without fighting each other.

---

## The I2C Protocol — What Happens on the Wire

A complete I2C transaction looks like this:

```
Master: START | SLA+W | data byte | data byte | ... | STOP
Master: START | SLA+R | ← data byte | ← data byte | ... | STOP
```

### START condition
SDA goes LOW while SCL is still HIGH. This is the signal to all devices on the bus:
"a transaction is beginning." Every device on the bus wakes up and listens.

### Address byte (SLA — Slave Address)
The first byte after START is always the **slave address + direction bit**:

```
bit 7  bit 6  bit 5  bit 4  bit 3  bit 2  bit 1  bit 0
└──────────────────────────┘                      └───┘
       7-bit device address                     R/W bit
                                            0 = Write (SLA+W)
                                            1 = Read  (SLA+R)
```

For the BH1750 at address `0x23`:
- Write: `(0x23 << 1) | 0` = `0x46`
- Read:  `(0x23 << 1) | 1` = `0x47`

The addressed slave pulls SDA LOW for one clock cycle to ACK (Acknowledge).
Every other device on the bus ignores the rest of the transaction.

### Data bytes
Each byte is followed by an ACK/NACK (Negative Acknowledge) from the receiver.
- In **write mode**: slave ACKs each byte it accepts.
- In **read mode**: master ACKs all bytes except the last (NACK on last byte
  tells the slave "stop, I have enough").

### STOP condition
SCL goes HIGH, then SDA goes HIGH. This releases the bus.

## ATmega328P TWI Peripheral

The hardware I2C controller on the ATmega328P is called **TWI** (Two Wire Interface).
It handles the low-level bit-banging: clock generation, start/stop conditions, shift
register, ACK detection — all in hardware. Your job is to configure it and check status.

### The Four TWI Registers

| Register | Name              | Purpose                                              |
|----------|-------------------|------------------------------------------------------|
| `TWBR`   | Bit Rate Register | Sets the SCL clock frequency                         |
| `TWCR`   | Control Register  | Sends commands to the peripheral (START, STOP, etc.) |
| `TWSR`   | Status Register   | Reports the outcome of the last operation            |
| `TWDR`   | Data Register     | Byte to transmit or last byte received               |

---

## TWBR — Setting the Clock Frequency

```
F_SCL = F_CPU / (16 + 2 * TWBR * prescaler)
```

For 100 kHz standard-mode I2C at 16 MHz:

```
100,000 = 16,000,000 / (16 + 2 * TWBR * 1)
16 + 2 * TWBR = 160
TWBR = 72
```

```cpp
TWSR &= ~0x03;  // prescaler bits PS1:PS0 = 00 → prescaler = 1
TWBR = 72;
```

The prescaler is encoded in `TWSR` bits 1:0:

| PS1 | PS0 | Prescaler |
|-----|-----|-----------|
|  0  |  0  |     1     |
|  0  |  1  |     4     |
|  1  |  0  |    16     |
|  1  |  1  |    64     |

---

## TWCR — Control Register

Every TWI operation is triggered by writing to `TWCR`. The key bits:

| Bit | Name    | Meaning                                                     |
|-----|---------|-------------------------------------------------------------|
|  7  | `TWINT` | **Interrupt flag.** Hardware sets it when an operation completes. Write `1` to clear it and start the next operation. |
|  6  | `TWEA`  | Enable ACK. When set, hardware automatically sends ACK after receiving a byte. Clear it for the last byte (NACK). |
|  5  | `TWSTA` | Generate a START condition on the next write.               |
|  4  | `TWSTO` | Generate a STOP condition. Hardware clears this bit automatically. |
|  2  | `TWEN`  | Enable the TWI peripheral. Must always be set.              |

**Critical rule:** To start any operation, you write `TWCR` with `TWINT = 1` (to
clear the flag) alongside whatever command bits you need. The peripheral then
executes and sets `TWINT` again when done. This is the entire polling loop.

---

## TWSR — Status Register

After `TWINT` is set (operation complete), read `TWSR & 0xF8` to get the status.
The lower 3 bits are the prescaler — mask them out.

### Master Transmitter mode (write transactions)

| Status | Meaning                              |
|--------|--------------------------------------|
| `0x08` | START condition transmitted          |
| `0x10` | Repeated START transmitted           |
| `0x18` | SLA+W transmitted — slave ACK'd      |
| `0x20` | SLA+W transmitted — slave NACK'd     |
| `0x28` | Data byte transmitted — slave ACK'd  |
| `0x30` | Data byte transmitted — slave NACK'd |

### Master Receiver mode (read transactions)

| Status | Meaning                                      |
|--------|----------------------------------------------|
| `0x40` | SLA+R transmitted — slave ACK'd              |
| `0x48` | SLA+R transmitted — slave NACK'd             |
| `0x50` | Data byte received — master sent ACK         |
| `0x58` | Data byte received — master sent NACK (last) |

**The most important thing to internalize:** `SLA+W` and `SLA+R` have **different
status codes** even though they both just send an address byte. The hardware
distinguishes them because the direction bit changes the mode of the peripheral.
This is what caused the first bug in this project's BH1750 driver — see
[Bugs section](#bugs-found-while-building-this-driver) below.

---

## Polling vs Interrupts

The TWI peripheral can signal completion via:
- **Interrupt** (`TWIE` bit in `TWCR`): the ISR (Interrupt Service Routine) fires when `TWINT` is set.
- **Polling**: you spin in a loop checking `TWCR & (1 << TWINT)`.

This driver uses **polling**. For a FreeRTOS task with a mutex protecting the bus,
polling is simpler and the transaction is short enough (~360 µs for a 2-byte read
at 100 kHz) that spinning is not a problem.

---

## Timeout — Why Not Timer1?

The polling loop needs a timeout so a stuck bus doesn't block the task forever.
The obvious choice on AVR is Timer1 (`TCNT1`) — but in this project, **that's wrong**.

FreeRTOS (feilipu/Arduino_FreeRTOS_Library) uses **Timer1 in CTC (Clear Timer on Compare) mode** as its
tick source. In CTC mode, `TCNT1` resets to 0 on every compare match — which
happens every millisecond. The reset is triggered by the Timer1 compare ISR.

A subtraction-based elapsed-time check with `TCNT1`:

```cpp
// BROKEN in a FreeRTOS context
uint16_t start = TCNT1;   // e.g. start = 200
// ... FreeRTOS tick fires, TCNT1 resets to 0 ...
uint16_t elapsed = (uint16_t)(TCNT1 - start);  // (0 - 200) = 65336 in uint16_t
// 65336 >= timeout → false timeout triggered immediately
```

Every TWI operation fails the moment FreeRTOS resets the timer.

**Note:** The DHT22 driver also uses `TCNT1` and works fine because it wraps
all its reads inside `taskENTER_CRITICAL()`, which disables interrupts — including
the Timer1 compare ISR. With interrupts disabled, FreeRTOS cannot reset `TCNT1`.
The BH1750 driver runs in a normal FreeRTOS task without disabling interrupts, so
the tick can fire at any time during the polling loop.

### The fix — loop counter

```cpp
#define TWI_TIMEOUT_ITERS 10000U

static bool twi_wait() {
    uint16_t count = TWI_TIMEOUT_ITERS;
    while (!(TWCR & (1 << TWINT))) {
        if (--count == 0) {
            return false;
        }
    }
    return true;
}
```

At 16 MHz, a TWI byte at 100 kHz takes ~1440 CPU cycles. With a simple 4-cycle
loop body, that is ~360 iterations per byte. 10 000 iterations gives a 27× safety
margin for a stuck bus, without depending on any timer.

The downside: the "timeout" is CPU-time based, not wall-time based. If the task
is context-switched out mid-loop, the counter pauses. In practice this is fine —
when the task resumes, `TWINT` is almost certainly already set (the hardware kept
running while the task slept), so the loop exits immediately.

## The Driver — Operation by Operation

### 1. Send START

```cpp
static bool twi_start() {
    TWCR = (1 << TWINT) | (1 << TWSTA) | (1 << TWEN);
    if (!twi_wait()) {
        return false;
    }
    uint8_t status = TWSR & 0xF8;
    return (status == 0x08 || status == 0x10);
}
```

- Clear `TWINT` + set `TWSTA` + `TWEN` → peripheral generates a START condition.
- After `TWINT` is set again, check for `0x08` (first START) or `0x10` (repeated START).
- A repeated START is a START issued without a preceding STOP — used when you need
  to switch from write to read in the same transaction (master regains bus control).

### 2. Send SLA+W and data bytes

```cpp
static bool twi_write(uint8_t data) {
    TWDR = data;
    TWCR = (1 << TWINT) | (1 << TWEN);
    if (!twi_wait()) {
        return false;
    }
    uint8_t status = TWSR & 0xF8;
    return (status == 0x18 || status == 0x28);
}
```

- Load `TWDR` with the byte, clear `TWINT` + enable TWI → hardware shifts it out.
- `0x18`: address byte with write bit sent, slave ACK'd.
- `0x28`: data byte sent, slave ACK'd.
- Both cases indicate success; the same code handles both.

### 3. Send SLA+R — READ this carefully

```cpp
static bool twi_addr_r() {
    TWDR = (BH1750_ADDR << 1) | 1;
    TWCR = (1 << TWINT) | (1 << TWEN);
    if (!twi_wait()) {
        return false;
    }
    return (TWSR & 0xF8) == 0x40;
}
```

Sending the address byte with the **read bit** (`| 1`) looks identical to sending
a write address byte — same `TWDR` write, same `TWCR` write. But the hardware
knows the difference: it switches to Master Receiver mode and returns status
`0x40` instead of `0x18`.

If you reuse `twi_write()` for this — as the original BH1750 driver did — the
status check `(status == 0x18 || status == 0x28)` always fails on `0x40`, and
every read returns false. The sensor never responds.

**This is not obvious** — nothing in the control register write is different. The
distinction only exists in the status code you check afterwards.

### 4. Receive a byte with ACK (more to follow)

```cpp
static bool twi_read_ack(uint8_t *out) {
    TWCR = (1 << TWINT) | (1 << TWEA) | (1 << TWEN);
    if (!twi_wait()) {
        return false;
    }
    if ((TWSR & 0xF8) != 0x50) {
        return false;
    }
    *out = TWDR;
    return true;
}
```

- `TWEA = 1` tells the hardware: after receiving this byte, send ACK to the slave.
  The slave interprets ACK as "keep sending."
- After `TWINT` sets, the received byte is in `TWDR`.
- Status `0x50` = data received and ACK sent.

### 5. Receive the last byte with NACK

```cpp
static bool twi_read_nack(uint8_t *out) {
    TWCR = (1 << TWINT) | (1 << TWEN);
    if (!twi_wait()) {
        return false;
    }
    if ((TWSR & 0xF8) != 0x58) {
        return false;
    }
    *out = TWDR;
    return true;
}
```

- `TWEA = 0` (not set) → hardware sends NACK. The slave stops transmitting.
- Status `0x58` = data received and NACK sent.
- You must read the last byte this way; if you ACK everything, the slave keeps
  driving SDA and you cannot generate a STOP cleanly.

### 6. Send STOP

```cpp
static void twi_stop() {
    TWCR = (1 << TWINT) | (1 << TWSTO) | (1 << TWEN);
}
```

- `TWSTO = 1` → hardware generates the STOP condition on the bus.
- The hardware clears `TWSTO` automatically; you do NOT wait for `TWINT` after STOP.
  There is no STOP status code to check.

## BH1750 Command Sequence

The BH1750FVI accepts single-byte opcodes over I2C. There is no register address —
each command byte is a complete instruction.

| Opcode | Meaning                                          |
|--------|--------------------------------------------------|
| `0x01` | Power ON                                         |
| `0x00` | Power OFF                                        |
| `0x10` | Continuous High Resolution Mode (1 lux, 120 ms)  |
| `0x11` | Continuous High Resolution Mode 2 (0.5 lux)      |
| `0x13` | Continuous Low Resolution Mode (4 lux, 16 ms)    |

### Init sequence

```
START → SLA+W (0x46) → 0x01 (Power ON) → STOP
wait 10ms
START → SLA+W (0x46) → 0x10 (Continuous HR mode) → STOP
wait 180ms  ← first measurement must complete before first read
```

### Read sequence (every 1 second)

```
START → SLA+R (0x47) → read MSB (ACK) → read LSB (NACK) → STOP
```

### Converting raw value to lux

The BH1750 returns a 16-bit value, big-endian (MSB (Most Significant Byte) first):

```cpp
uint16_t raw = ((uint16_t)msb << 8) | lsb;  // msb = Most Significant Byte, lsb = Least Significant Byte
float lux = raw / 1.2f;
```

The 1.2 divisor comes from the datasheet: the sensor counts in units of
`(1/1.2)` lux by design. At full scale (`0xFFFF = 65535`), the result is
`65535 / 1.2 = 54612 lux`.

## Why Not Wire.h?

`Wire.h` is Arduino's C++ wrapper around the exact same TWI hardware.
Reasons to bypass it in this project:

1. **Removes a library dependency** — `claws/BH1750` depended on `Wire`, which
   pulled in ~600 bytes of overhead (ring buffers, ISR vectors) for something
   that can be implemented in ~80 bytes.

2. **No hidden ISR conflicts** — `Wire.h` by default uses the TWI interrupt
   (`TWIE`) to drive transactions asynchronously. This ISR can fire at unexpected
   times and interact badly with FreeRTOS priorities.

3. **Full control over timeout behavior** — `Wire.setWireTimeout()` exists but
   works through its own ISR machinery. A loop-counter timeout is simpler,
   more predictable, and doesn't depend on timer configuration.

4. **Consistency** — DHT22 and PIR are already bare-metal. Bare-metal TWI
   completes the pattern: every driver in this project speaks directly to hardware.

## Bugs Found While Building This Driver

### Bug 1 — Timer1 false timeouts

**Symptom:** BH1750 never appeared in serial output. `bh1750_read()` silently
returned false on every call.

**Cause:** First implementation used `TCNT1` for the polling timeout. FreeRTOS
resets `TCNT1` every millisecond via its Timer1 CTC ISR. Every reset caused the
unsigned subtraction `(TCNT1 - start)` to wrap to ~65000, instantly exceeding the
timeout threshold.

**Fix:** Replace timer-based timeout with a loop counter (`TWI_TIMEOUT_ITERS`).

---

### Bug 2 — Wrong status code for SLA+R

**Symptom:** Same as Bug 1 — no LIGHT readings, `bh1750_read()` always false.

**Cause:** `bh1750_read()` passed the read-direction address byte (`SLA+R`)
through `twi_write()`, which checked for statuses `0x18` and `0x28`. The hardware
returned `0x40` (Master Receiver address ACK) — a different code — so the check
always failed.

**Fix:** Separate `twi_addr_r()` function that checks for `0x40`.

**Lesson:** The TWI status code table has two completely separate sections for
write-direction and read-direction address bytes. They use the same mechanical
operation but different status codes. Always check both columns of the datasheet.

## Full Status Code Quick Reference

| Code   | Mode | Event                                    |
|--------|------|------------------------------------------|
| `0x08` | MT   | START transmitted                        |
| `0x10` | MT   | Repeated START transmitted               |
| `0x18` | MT   | SLA+W transmitted, ACK received          |
| `0x20` | MT   | SLA+W transmitted, NACK received         |
| `0x28` | MT   | Data byte transmitted, ACK received      |
| `0x30` | MT   | Data byte transmitted, NACK received     |
| `0x40` | MR   | SLA+R transmitted, ACK received          |
| `0x48` | MR   | SLA+R transmitted, NACK received         |
| `0x50` | MR   | Data byte received, ACK returned         |
| `0x58` | MR   | Data byte received, NACK returned (last) |

MT = Master Transmitter mode (write transaction)  
MR = Master Receiver mode (read transaction)

## Complete Driver at a Glance

```
bh1750_init():
  TWBR=72, TWSR prescaler=1  → 100 kHz
  START → SLA+W → POWER_ON → STOP → wait 10ms
  START → SLA+W → CONT_HR  → STOP → wait 180ms

bh1750_read():
  START → SLA+R → read MSB (ACK) → read LSB (NACK) → STOP
  lux = ((MSB << 8) | LSB) / 1.2
```

Each arrow is one TWI primitive. Each primitive:
1. Writes `TWCR` to trigger the operation
2. Polls `TWCR & TWINT` until the hardware is done
3. Checks `TWSR & 0xF8` against the expected status code
4. Returns false on timeout or unexpected status
